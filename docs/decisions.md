# Decisions

## Goを採用した理由

Goは標準ライブラリだけでもHTTP APIを実装しやすく、シンプルで保守しやすいバックエンドを作りやすいため採用しました。

また、将来的にAPIサーバー、バッチ処理、GitHub連携処理などへ拡張する場合も、単一バイナリとして扱いやすく、デプロイや運用を簡潔にできます。

## MVPではGitHub連携を後回しにした理由

最初に検証したい価値は、コードを入力してレビュー結果を得る体験そのものです。

GitHub連携を先に実装すると、OAuth、リポジトリ権限、Pull Request取得、コメント投稿などの実装範囲が大きくなります。MVPではレビュー体験の検証を優先し、GitHub連携は後続フェーズで対応します。

## MVPでは認証機能を実装しない理由

初期段階では個人利用を前提とし、レビューAPIと履歴保存の基本動作を優先します。

認証機能はユーザー管理、セッション管理、権限制御などの設計が必要になるため、MVPの検証範囲からは外します。チーム利用や公開運用を見据える段階で追加します。

## まずは個人利用を前提に設計する理由

個人利用に絞ることで、レビュー入力、レビュー結果表示、履歴保存、Codex向けプロンプト生成という中核機能に集中できます。

最初からチーム利用や権限管理を前提にすると、設計と実装が重くなります。まずは小さく動く体験を作り、利用価値を確認してから機能を広げます。
